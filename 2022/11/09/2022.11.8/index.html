# PPP
# 数据链路层协议：

​ 局域网：Ethernet，IEEE802.3

​ 广域网：PPP，HDLC 高级数据链路控制协议，FR 帧中继，ATM 信元传输 ，X.25

# 广域网设备角色分类：

​ CE: 用户边缘设备，属于用户，用来和广域网进行连接，作为私网的出口网关设备

​ PE: 运营商边缘设备，属于运营商，需要和私网的 CE 连接

​ P: 运营商骨干设备，属于运营商，不需要和用户边缘设备连接

# ppp 特点：

​ 支持全双工的同步和异步传输

​ 支持数据链路层协商，LCP

​ 支持对链路进行认证

​ 支持网络层协商，NCP

​ 良好的扩展性，比如 PPPoE

# PPP 报文格式：

​ Flag：标识数据帧的起始和结尾，用于区分不同的数据帧

​ Address：用来唯一标识对端设备，但是由于点到点网络中，对端设备只有一台，所有没有使用该字段标识对端的数据链路层地址，全部填充 1，对应十六进置 F

​ Control：无意义，固定取值 0x03

​ Address+Control 固定取值为 0xFF03，能够用来标识 PPP 封装类型

​ Protocol：协议字段，用来标识后面数据使用的封装协议

​ information：信息，PPP 报文封装的载荷

​ LCP 封装格式：

​ Code：代码，用来标识 LCP 报文类型

​ identifier：标识同一种类型的不同报文

​ Length：LCP 报文长度

​ Data：协商数据，采用 TLV 结构封装

​ T: TYPE，类型

​ L:Length，长度

​ V: Value ， 取值

​ FCS：用来校验数据帧的完整性

​ Flag：同上

# PPP 的工作流程：

1.LCP, 数据链路层协商：用于数据链路层参数的协商，建立，维护和拆除链路层连接

  1. LCP 报文类型：
    1. 配置请求报文：Request，用来请求建立数据链路层的连接，携带协商参数
  2. 配置确认报文：ACK，用来对请求进行确认回应，表示参数协商成功

  3. 配置否认报文：NAK，用来对请求进行否认回应，表示参数协商失败，携带协商失败的参数

  4. 配置拒绝报文：Reject，用来对请求进行拒绝回应，表示参数不识别，携带无法识别的参数

  1. LCP 协商参数：

    1. MRU: 最大接收单元，接口单位时间内能够接收的最大数据量，默认 1500B，单位字节
2. 认证类型:不认证，PAP，CHAP，默认不认证

3. 魔术字:魔术字是一个随机的取值，用来防止环路 ，如果收到的LCP报文的魔术字和自己产生的魔术字相同，则认为产生环路，丢弃报文
  1. LCP 协商流程：数据链路层的连接是单向的，双方都需要建立
1. 双方会发送配置请求报文，携带LCP协商参数（MRU,认证，魔术字）

2. 收到对方的配置请求后，检查其中的携带参数：

   ①如果参数协商成功，回应ACK，表示参数协商成功

   ②如果参数协商失败，回应NCK,表示参数协商失败，同时会携带上协商失败的参数

   ③如果参数不识别，回应Reject，表示参数不识别，同时会携带上不识别的参数

3. 收到对方回应后：

​	①如果收到ACK，表示数据链路层连接建立完成

​	②如果收到NAK，表示协商失败，根据NAK中指定的协商失败参数修改本地对应参数，然后再进行配置请求重新建立链路层连接

​	③如果收到Reject，表示参数不识别，就需要删除Reject中指定的参数再重新发生配置请求建立链路层的连接

2. 认证：使用 LCP 协商的认证类型对链路进行认证，PAP 或者 CHAP

​ 不认证：默认不进行认证

​ PAP：密码认证协议

​ 特点：被认证方发起，两次交互完成，密码明文携带，不安全不可靠不推荐

​ 流程：

​ 1. 被认证方将自己的用户名和密码发送给我们的认证方，密码明文发送

​ 2. 认证方收到之后，根据其中携带的用户名，查找数据库中该用户对应的密码，如果密码和报文携带的一样，就说明认证成功，回应 ACK；如果不一致，就说明认证失败，回应 NAK

​ 配置：

​ 认证方:

​ 1. 接口指定 PPP 认证模式

​ [R1-Serial4/0/0]ppp authentication-mode pap

​ 2. 再 AAA 下创建对应的用户和密码，并指定其为 PPP 服务

​ [r1]aaa

​ [r1-aaa] local -user haha passwrod 123

​ [r1-aaa] local -user haha service-type ppp

​ 被认证方：

​ 1. 接口配置用户名和密码

​ [r2-s4/0/0] ppp pap local–user haha password cipher huawei123

​

​ CHAP：挑战握手认证协议

​ 特点：认证方发起，三次交互，密码是加密传输，安全可靠推荐使用

​ 流程：

​ 1. 认证方向被认证方发送挑战信息，其中携带 ID，随机值，用户名（可选）

​ 2. 被认证方收到后，将本地的密码，ID 以及随机值进行 Hash 计算，得到摘要值，将本地用户名和摘要值以及 ID 响应给认证方

​ 3. 认证方收到后，根据携带的用户名查找本地数据库，找到对应的密码，将密码，ID 以及随机值同样进行 Hash 计算，得到摘要值，将自己计算的摘要值和报文中携带的摘要值进行对比：

​ 如果一致则说明密码正确，回应认证成功

​ 如果不一致则说明密码错误，回应认证失败

​ 配置：

​ 认证方:

​ 1. 接口指定 PPP 认证模式

​ [R1-Serial4/0/0]ppp authentication-mode chap

​ 2. 再 AAA 下创建对应的用户和密码，并指定其为 PPP 服务

​ [r1]aaa

​ [r1-aaa] local -user haha passwrod 123

​ [r1-aaa] local -user haha service-type ppp

​ 被认证方：

​ [r2-Serial4/0/0] PPP chap user haha

​ [r2-Serial4/0/0] PPP chap password cipher huawei123

3.NCP，网络层协商：协商使用的网络层协议已经网络逻辑地址

​ IPCP

​ NCP 报文分类：

  1. 配置请求报文：Request，用来请求建立的连接，携带协商参数
  2. 配置确认报文：ACK，用来对请求进行确认回应，表示参数协商成功
  3. 配置否认报文：NAK，用来对请求进行否认回应，表示参数协商失败，携带协商失败的参数
  4. 配置拒绝报文：Reject，用来对请求进行拒绝回应，表示参数不识别，携带无法识别的参数

工作流程：

​ 静态 NCP 协商：双方都配置了 IP 地址

​ 1. 双方向对方发送配置请求，其中携带 IP 地址用于协商

​ 2. 收到对方的配置请求后，检查携带的 IP 地址是否合法，如果合法回应 ACK，不合法就回应 NAK

​ ppp 判断 IP 地址是否合法：只要 IP 地址不是自己使用的地址，也就是无冲突就算合法；

​ 注意：如果配置请求携带的 IP 地址为 0.0.0.0 也会认为不合法

​ IPCP 动态协商：一方没有配置 IP 地址

​ 1. 没有配置 IP 地址的一方会发送配置请求报文，其中携带的 IP 地址 0.0.0.0

​ 2. 另一方收到后，检查其中携带的 IP 地址，发现是 0.0.0.0，不合法，就会回应 NAK，其中携带上分配给对方使用的 IP 地址

​ 3. 没有配置 IP 地址的一方收到 NAK 后，会使用其中携带的 IP 地址，然后再发送配置请求，其中的 IP 地址新获取的 IP 地址

​ 4. 另一方收到后，检测 IP 地址，合法回应 ACK，不合法 NAK

​ 配置：

​ 配置 IP 地址的一方：

​ [r1-serial4/0/0]remote address 10.1.12.2

​ 没有配置 IP 地址的一方:

​ [R2-Serial4/0/0]ip address ppp-negotiate

MTU: 最大传输单元，接口单位时间（1s）内能够传输的最大数据量，默认 1500，单位字节

# PPPOE

​ pppoe , 以太网承载 PPP 协议

​ 背景：运营商需要对上网的用户进行认证和计费，这是 PPP 能够实现的，但是如果使用 PPP 的话，一台 CE 就需要连接到一台 PE 的一个接口，成本高，不现实，运营商希望多台 CE 能够接入到同一台 PE 的一个接口上，这是 Ethernet 能够实现的，但 Ethernet 又不支持认证和计费，所有产生 PPPoE，结合 PPP 和以太网的优点

# PPPOE 作用：

​ 将 PPP 数据帧封装到 Ethernet 数据帧的有效载荷部分，结合 PPP 的认证，计费和 Ethernet 多路访问的优点，实现多台 CE 连接到同一台 PE 的一个接口进行认证和计费，节约成本

​ PPPoE 采用 C/S 架构，客户端 / 服务器

# PPPOE 报文格式:

​ 封装格式：

​ D.MAC|S.MAC|TYPE| PPPOE 头部 | PPP 数据帧 | FCS

​ PPPoE 头部字段：

​ Version：标识版本，固定 0X01

​ TYPE: 固定 0X01

​ code：标识 PPPoE 报文类型

​ Session：PPPoE 会话 ID，和 S.MAC,D.MAC 一起用来唯一标识一个 PPPoE 会话

​ Length：ppp 数据帧的长度，不包括 PPPoE 报文字段长度

​ Ethernet 的 Type 取值：标识 PPPoE 报头

​ 发现阶段：0x8863

​ 会话阶段：0x8864

​ 终结阶段：0x8863

# PPPoE 报文类型：

​ PADI: 初始化报文，客户端广播发送，用来寻址 PPPoE 服务器

​ PADO: 服务报文，服务器单播回应客户端，携带服务器的 MAC 地址

​ PADR: 请求报文，客户端单播发送给服务器请求建立 PPPoE 的会话，并请求获取 PPPoE 会话 ID

​ PADS: 会话确认报文，服务器单播发送给客户端，携带 PPPoE 会话 ID

​ PADT: 终止报文，在 PPPoE 会话建立完成后，任何一方都可以随时通过单播发送该报文断开 PPPoE 会话连接

# PPPoE 工作流程：

​ 发现阶段：客户端发现服务器，建立 PPPoE 会话，获取 PPPoE 会话 ID

  1. 客户端广播发送 PADI，用来寻址服务器

  2. 服务器收到 PADI，先检查自己能够提供对应的服务（有无地址可分配），如果能提供服务的话，就会单播回应客户端 PADO 报文，携带服务器的 MAC 地址

  3. 客户端收到 PADO 后，优先选择先收到的进行回应，单播回应服务器 PADR 报文，请求建立 PPPoE 会话，并请求获取 PPPoE 会话 ID

  4. 服务器收到 PADR 后，建立 PPPoE 会话连接，并单播回应客户端 PADS 报文，携带 PPPoE 会话 id，后续交互的所有 PPPoE 报文都会携带上该会话 ID

  5. 客户端 PADS 后，表示 PPPoE 会话建立完成，双方都获取到对方的 MAC 地址以及 PPPoE 会话 ID, 发现阶段完成

    备注：DHCP 广播发送 Request 报文的原因：

    ​ ①向服务器请求 IP 地址

    ​ ②拒绝其他服务器，让其它服务器能够及时回收 IP 地址，避免浪费

    ​ PPPoE 单播发送给 PADR 的原因：

    ​ PADO 报文并不会携带 IP 地址，只会携带 MAC 地址，并不会造成地址浪费问题

​ 会话阶段：分为 PPP 协商过程以及后续的 PPP 数据传输

​ PPP 协商：

​ lcp：协商数据链路层参数

​ 认证：PPPoE 一定会进行认证流程，CHAP

​ ncp：PPPoE 客户端通过 IPCP 动态协商从服务器获取公网 IP 地址

​ PPP 数据传输：获取到公网 IP 地址后，用户就可以正常上网

​ 终止阶段：PPPoE 会话建立完成后，任何一方都可以随时通过单播发送对方 PADT 报文来终止 PPPoE 会话

配置：
1、在 AAA 中创建对应的宽带账号和密码，用于 PPPoE 拨号
[r2] aaa
[r2-aaa]local-user 123456 password cipher huawei123
创建宽带账号以及对应的密码
[r2-aaa] local-user 123456 service-type ppp
配置用户为 PPP 服务
2、创建公网地址池，用来分配给客户端
[r2] ip pool pppoe
[r2-ip-pool-pppoe]network 10.1.12.0 mask 24
3、创建虚拟模板接口
[r2] interface Virtual-Template 1
[r2-Virtual-Template1]ip address 10.1.12.1 24
[r2-Virtual-Template1]remote address pool pppoe
配置为客户端分配地址使用的地址池
[r2-Virtual-Template1] ppp authentication-mode chap
配置认证模式为 CHAP
4、将虚拟模板接口绑定到物理接口上
[r2-GigabitEthernet0/0/0] pppoe-server bind virtual-template 1

客户端配置：
1、创建拨号规则
[r1] dialer-rule
[r1-dialer-rule]dialer-rule 1 ip permit
允许所有 IP 流量进行拨号
2、创建虚拟拨号接口
[r1] interface Dialer 1
[r1-Dialer1]dialer user dianxin
配置拨号接口的用户标识
[r1-Dialer1] dialer-group 1
配置拨号组
[r1-Dialer1] dialer bundle 222
配置拨号接口绑定到物理接口的编号
[r1-Dialer1] ip address ppp-negotiate
IP 地址通过 PPP 的 IPCP 协商获取
[1-Dialer1] ppp chap user 123456
配置 CHAP 认证使用的宽带账号
[r1-Dialer1] ppp chap password cipher huawei123
配置 CHAP 认证使用的宽带密码
3、将虚拟拨号接口绑定到物理接口上
[r1-G0/0/0] pppoe-client dial-bundle-number 222

客户端查看：
[r1]display pppoe-client session summary
服务器查看：
[r2]display pppoe-server session all
