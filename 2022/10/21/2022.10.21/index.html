# 传输层：

​ 用于建立端到端的连接

​ 再应用层数据前加上端头，数据称为数据段

# TCP：

​ 面向连接的安全可靠的传输层协议，

Source port (16) Destinationport(16)
Sequencenumber(32)
Acknowledgement number (32)
Header length (4) Reserved (6) Control bits (6) Window (16)
Checksum (16) Urgent (16)
Options
Data (varies)

​ TCP 头部参数：

​ 源端口号（Source port ）：用于标识发送者产生此数据的应用

​ 目的端口号（ Destination port ）：用于标识接收者最终交由的应用的那个用于进行处理

​ 序列号（Sequence number ）：标识发送者发送的第几个数据段

​ 确认序列号（Acknowledgement number）：用于接收者确认收到发送者发送的数据段

​ 头部长度（Header length）：用于标识 TCP 的头部长度，因为 TCP 的头部中存在 option（可选项）为不定长

​ 保留位（Reserved）：无意义，全为 0

​ 控制位（Control bits ）：用于特殊的作用

​ SYN：请求建立连接位

​ ACK：确认位

​ FIN：请求断开连接位

​ R：请求重新建立 TCP 连接

​ U：紧急指针位

​ 窗口（Window ）：用于标识接收者最大能够接收缓存的数据单位

​ 校验和（ Checksum ）：检验数据段的完整性

​ 紧急指针位（ Urgent ）：用于表示该数据需要紧急交由应用层协议进行处理，需要配合控制中的 U 位使用。

​ 可选项（ Options ）：用于携带一些可选参数，例如认证

​ 数据（Data (varies) ）：应用层产生的数据

# UDP 头部参数：

​ 源端口号（Source port ）：用于标识发送者产生此数据的应用

​ 目的端口号（ Destination port ）：用于标识接收者最终交由的应用的那个用于进行处理

​ 长度（Length）：标识数据段的长度

​ 校验和（ Checksum ）：检验数据段的完整性

​ 数据（Data（if any））：应用层产生的数据

# 端口号：

​ 用于标识应用层协议

​ 端口号范围：0-65535

​ 分为知名端口号和非知名端口号

​ 知名端口号：一般被知名的应用层协议所使用

​ 范围：0-1023

​ 例如：HTTP：80

​ HTTPS：443

​ 非知名端口号：可以个人使用，未被分配

​ 范围：1024 及以上都是

​

# TCP 三次握手:

​ SYN (用于请求建立 TCP 连接)

​ PC1<----------->PC2

​ 1.PC1 发送 SYN 置位的 TCP 报文，用于请求和 PC2 建立 TCP 连接，序列号 a (随机生成的)，确认序列号为 0（PC2 在此之前没有收到过 PC2 的数据)

​ 2.PC2 向 PC1 回应一个 SYN 和 ACK (确认 PC1 发送的 SYN 请求）置位的 TCP 报文，序列号 b（随机产生的)，确认序列号位 a+1 (1, 确认收到了 PC1 发的序列号为 a 的数据包，2 要求 PC1 下次发送数据包的序列号为 a+1)

​ 3.PC1 向 PC2 发送 ACK 置位的 TCP 报文，序列号 a+1，确认序列号为 b+1

# 序列号和确认序列号：

​ 用于保证数据传递的有序性和可靠性

​ 采用序列号 + 1 或者 + 数据长度的方式保证数据 包传递的有序性

​ 采用确认序列号对对方数据包中序列号 + 1 的方式保证数据包传递的可靠性

# TCP 滑动窗口：

​ 再两台主机建立 TCP 连接时，会将自己的窗口大小告知对方，等到 TCP 连接建立完成后，知道对方窗口大小，一次性传输的数据不会超过对方的窗口大小，当窗口被占用时，主机会将 TCP 报文中的窗口位降低，告知对方最大还允许发送的数据，当主机处理完一部分数据后，将窗口进行释放，同时也会将新的窗口大小通知对方

# TCP 重传机制：

​ PC1<--------------->PC2

​ 1. 当 PC1 收到 PC2 发送的数据时，中间缺失某个序列号的数据

​ 2. 那么 PC1 只会对缺失的数据前有序的流量进行确认，后续到达的数据不进行确认

​ 3.PC2 收到 PC1 发送的确认报文后，发送只对前面发送的数据进行了确认，启动重传，对后续未确认的数据进行重传

# TCP 四次挥手：

​ PC1<--------------->PC2

​ 1.PC1 请求断开 TCP 连接，发送 FIN（请求断开 TCP 连接）,ACK (确认位) 置位的 TCP 报文，用于请求断开 TCP 连接，序列号 a，确认序列号 b‘

​ 2.PC2 回应 ACK 置位的 TCP 报文，用于确认和 PC1 断开单项的 TCP 连接，序列号 b，确认序列号 a+1

​ 3.PC 发送 FIN（请求断开 TCP 连接）,ACK (确认位) 置位的 TCP 报文，用于请求断开 TCP 连接，序列号 b，确认序列号 a+1

​ 4.PC1 回应 ACK 置位的 TCP 报文，用于确认和 PC2 断开 TCP 单向连接，序列号 a+1，确认序列号 b+1

# TCP 应用场景：

​ 对于安全性，可靠性要求比较高的流量可以使用 TCP 进行发送

# UDP 特点：

​ 1. 头部短 2. 面向无连接

# UDP 应用场景：

​ 对于安全性要求不高，但是数据比较大

​ 1. 视频流量

​ 2. 电话流量

​ 3. 游戏流量

# TCP 于 UDP 的对比：

​ 1.TCP 安全可靠：三次握手，四次挥手，滑动窗口，重传机制，序列号，确认序列号。

​ 2.UDP 面向无连接，安全必须通过其他层次协议进行保证 。

# 应用层：

​ 数据称之为 PDU, 协议数据单元

​ HTTP: 简单文本传输协议，端口号 80

​ HTTPS：超文本传输协议，端口号 443

​ FTP：文件传输协议， 端口号 20/21

​ SSH：安全外壳协议，端口号 22

​ Telent ：远程登录管理协议， 端口号 23

​

# 数据封装：

​ 1. 应用层产生数据，始称 PDU，协议数据单元

​ 2. 应用层 将数据交由传输层进行处理

​ 3. 传输层再 PDU 前加上端头，包含端口号，数据称之为数据段

​ 4. 传输层将数据交由网络层进行处理

​ 5. 网络层在数据段前加上 IP 头，包含 IP 地址，数据称之为数据包

​ 6. 网络层将数据交由数据链路层 处理

​ 7. 数据链路层在数据包前加上帧头，在数据包后加上帧尾，包含 MAC 地址，数据称之为数据帧

​ 8. 数据链路层将数据交由物理层进行处理

​ 9. 物理层将数据转化为比特流转发出去

# 数据解封装：

​ 1. 物理层收到一个比特流，将比特流转化为数据帧交由数据链路层进行处理

​ 2. 数据链路层收到数据帧后，

​ a）查看尾校验是否通过，通过则进行下一步，不通过则直接丢弃

​ b）查看帧头目的 MAC 地址是否单播，组播，广播

​ c）为单播时，是否为自身 MAC 地址，是接收处理，交由网络层，不是直接丢弃

​ d）为组播时，是否为自身所在组的 MAC 地址，是则接收处理，交由网络层，不是丢弃

​ e）为广播时，直接处理，交由网络层

​ 3，网络层收到数据包，

​ a）查看头部校验和，检验数据包完整性，通过则进行下一步，不通过直接丢弃

​ b）查看 IP 头部中的目的 IP 地址，为单播，组播，广播

​ c）为单播时，判断是否为自身的 IP 地址，时则直接交由传输层，不是，判断自身是否为主机或者路由器等三层设备

​ i . 为主机时，直接丢弃

​ ii . 为路由器等设备时，按照路由表进行转发

​ d）为组播时，判断是否自身所在 IP 地址，是则交由传输层，不是则丢弃

​ e）为广播时，直接交由传输层处理

​ 4. 传输层收到数据段，查看端口中的目的端口号，直接将数据交由对于的应用层协议进行处理

​ 5. 应用层，对数据进行处理，并且对源进行回应
